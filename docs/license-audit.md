## Compatibility analysis

**Apache-2.0 ↔ Apache-2.0** is trivially compatible: the same license terms apply. The patent grant in Apache 2.0 carries through.

**MIT ↔ Apache-2.0** is one-way compatible in the direction this project needs it: MIT-licensed code can be incorporated into an Apache-2.0-licensed project without re-licensing the MIT components. The project as a whole is distributed under Apache 2.0; the MIT components retain their MIT notices, which are preserved in `NOTICE`.

The reverse direction (Apache-2.0 code in an MIT project) has subtleties around the patent termination clause, but this project does not redistribute under MIT, so that case does not apply.

## Obligations triggered by the chosen license combination

### From Apache 2.0 (Section 4)

This project must:

1. **Include a copy of the Apache 2.0 license** in any redistribution. ✅ Provided in `LICENSE`.
2. **Preserve copyright, patent, trademark, and attribution notices** from upstream Apache-2.0 dependencies. ✅ Provided in `NOTICE` for `puppeteer` and `@google/genai`.
3. **State that files were modified**, where applicable. Not applicable — this project does not modify upstream dependency source code, it only depends on it.
4. **Provide a NOTICE file** with attributions if the upstream NOTICE files exist. ✅ Provided. (Upstream `puppeteer` and `@google/genai` do not currently ship NOTICE files of their own; the attribution in our NOTICE is sufficient.)

### From MIT (each MIT-licensed dependency)

The MIT license requires:

1. **Inclusion of the MIT license notice and copyright** in any redistribution of the MIT-licensed software.

This project does not bundle or redistribute the MIT-licensed dependencies in its own repository — they are installed by `npm install` from npm at the user's machine, and the dependencies' own `LICENSE` files are present in `node_modules/{package}/LICENSE`. No further action is required by this project's repository.

If any user produces a bundled distribution (e.g. an npm-published binary, a Docker image, a desktop app) that includes the MIT dependencies' compiled code, that bundle's distribution must include the MIT notice for each bundled MIT package. Standard practice is to ship a `THIRD_PARTY_LICENSES.txt` file generated by `license-checker`.

For attribution clarity, this project's `NOTICE` enumerates the MIT-licensed dependencies even though it is not strictly required by MIT. This is courtesy, not obligation.

## Transitive dependency caveat

This audit covers the six direct dependencies the user requested. Transitive dependencies (the dependencies of those dependencies) were not individually audited.

To verify transitive licenses, after `npm install` run:

```bash
npx license-checker --summary
npx license-checker --csv > third-party-licenses.csv
```

In practice, the npm ecosystem dominated by Node.js infrastructure libraries is overwhelmingly MIT, ISC, BSD-2/3, and Apache-2.0. Copyleft licenses (GPL, AGPL, LGPL) are exceptionally rare in this dependency graph. A scan is still recommended before any distribution event because the answer can change with version bumps.

## What this audit does not cover

- **Upstream API terms.** This project calls the Anthropic API, the Google Gemini API, the Buffer API, and the Meta Graph API. Those APIs have their own commercial terms (TOS, AUP, data processing agreements) that govern your use of them. The dependency licenses above only govern the SDK code that talks to those APIs, not the APIs themselves.
- **Trademarks.** Apache 2.0 Section 6 explicitly does not grant trademark rights. "Puppeteer", "Gemini", "Claude", "Buffer", "Instagram", and "Meta" are trademarks of their respective owners. Don't use them in your forked project's name.
- **Patents.** The Apache 2.0 dependencies grant a patent license per Section 3. The MIT dependencies do not include an explicit patent grant. If your use case has unusual patent exposure, consult a lawyer — this audit is not legal advice.

## Re-running this audit

Anyone forking should re-verify before publishing. Steps:

1. `npm install`
2. `npx license-checker --summary` — sanity check the distribution.
3. For each direct dependency, visit the npm page or `node_modules/{package}/package.json` and confirm the `license` field.
4. Update this file with the new audit date.

License changes between major versions are rare but not impossible. The most likely vector for a license-state change is a maintainer changing their mind on a hobby project — `better-sqlite3` and `zod` are essentially solo-maintained, and a re-licensing event is theoretically possible. If that happens, a fork running an old version is unaffected; a fresh `npm install` on a relicensed version would pull in the new terms.
